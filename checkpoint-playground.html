When an agent works, each step is saved as a commit on a shadow branch. The shadow branch forks from the current HEAD and stores full file snapshots + session metadata. It never touches your working branch.

Rewind restores your working directory to a previous checkpoint. Unlike git reset, it reads files directly from the checkpoint's git tree and writes them to disk, preserving your branch history.

How Rewind Works (Step by Step)

1. Read checkpoint commit tree from shadow branch
2. Compare files in tree vs. current working directory
3. Compute FilesToRestore — files from checkpoint to write
4. Compute FilesToDelete — untracked files created after checkpoint
5. Compute TrackedChanges — uncommitted tracked changes to revert
6. Write files from git tree to disk (no git reset)
7. Session metadata is preserved for reference
A 12-hex-char checkpoint ID creates a bidirectional link between your git commit and the session metadata. The ID is embedded as a trailer in your commit and used as a sharded directory path on the metadata branch.

Sharding Explained

The checkpoint ID a3b2c4d5e6f7 is split: first 2 chars become a directory prefix (a3/), remaining 10 become a subdirectory (b2c4d5e6f7/). This creates up to 256 shards (00-ff), preventing any single directory from holding too many entries.
